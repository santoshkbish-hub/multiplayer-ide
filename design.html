Collaborative Local AI Coding

The host laptop is the source of truth. Remote collaborators connect through a thin relay; the host app manages shared workspaces, agent runs, safety boundaries, and every publish to main.

Overview

How it fits together

  • A work session is the shared workspace: one file tree, one draft of work, and one safe place for agents to run. Multiple conversations can collaborate inside it.
  • A conversation is the user-facing chat lane: it has its own transcript and agent memory, but it can still work in the same shared workspace.
  • Clients never touch the host repo or terminal directly. The relay carries messages; the host app decides what can run and what can change.
  • Agents stay aware of each other through short activity updates and host-provided change notices, so they can see when another agent may affect the same files.
  • Publishing is a gated step: the host checks the changes, protects main, and serializes final merges.
Architecture

Components & trust boundary

Shared workspaces

Multiple agents in one workspace

The UI shows active work sessions. A user can resume an existing conversation or start another conversation inside the same workspace. Every agent sees the same file tree, so the product keeps each conversation aware of nearby work.

workspace

Shared working state

One work session has one file tree and one draft of work. When several conversations join it, they are collaborating in the same place, not in copies.

conversation

Separate chat context

Each conversation keeps its own transcript and agent memory. Starting another conversation does not create another workspace.

awareness

Coordination, not polling

Agents receive short updates about what others are doing and what changed on main. They can add their own concise updates before important edits.

Coordination

How agents stay out of each other’s way

Two people can each be steering an agent inside the same workspace. The system keeps every agent aware of what the others are doing, lets an agent ask a referee when something might overlap, and pauses cleanly when there is no good automatic answer. Nothing is locked by default — agents run side by side and only slow down when they actually bump into each other.

activity board

Constant awareness

A short summary of every other agent’s recent work and current plan is handed to each agent before its next step. It costs the agent nothing to read; it just shows up.

plan note

Say what you’re about to do

Before changing files, an agent posts a one-line plan: the goal and which files it expects to touch. Other agents see it on the activity board; the system uses it to spot overlap.

referee

Ask when in doubt

A separate read-only helper that an agent can ask any time it senses overlap. It looks at every active plan note and answers with one of a small set of suggestions.

Behind the scenes — who fires what

Three actors split the work. None of them watches all of it — that’s on purpose. Agents stay parallel because nobody is sitting between them deciding what they can do.

per-chat agent

Runs only while you’re prompting it

Posts plans and messages, asks the referee when something might overlap. When it steps aside, nothing is running on its side — the question is held until the wait is over. It never polls or schedules itself.

referee

Only thinks when asked

Reads the board, weighs the open plan notes, returns one of the five answers, then goes idle. Doesn’t watch agents, doesn’t push messages. A person can step in and answer in the referee’s place at any moment.

always-on watcher

Notices when things finish

The one piece always listening. When a turn ends, when a plan is marked done, when files are written — the watcher reacts. When a paused agent’s wait just resolved, this is what wakes it — with a short note explaining what changed.

The agent never decides “is it my turn yet?” — while paused, it isn’t doing anything. The watcher notices the wait condition resolving and re-fires the agent’s question with a wake-up note. The referee is never in the loop for wake-ups; it only gets asked when the agent itself wants advice. The channels are kept to the minimum that works: agent ↔ board (passive memory), agent ↔ agent (direct messages), agent → referee (asked questions), watcher → agent (wake-ups). Anything else would mean a long-running brain holding the system together, which is exactly what this design avoids.

A few things to keep in mind from the diagrams above. Agents are not serialized by default — two people prompting at the same moment both start working immediately. The only time work pauses is when an agent itself decides to step aside, or when two people are clearly asking for the same thing. Awareness arrives passively, between every batch of actions, so an agent doesn’t have to remember to check. And resuming a paused turn is never blind: the wake-up always comes with a short note explaining what changed while the agent was away.

Flow

How work moves through the system

Relay

The relay server

The relay is the live transport layer. Browsers connect to it from the LAN, and the host app keeps one long-lived connection open. The relay groups people by workspace, carries presence and chat events, and forwards agent updates back to the right conversations.

The relay does not own project state. If it restarts, users reconnect and the host app restores the visible history. File access, agent actions, and publish decisions stay on the host laptop.

⚠ Single point of failure

One relay node is exactly that — if it goes down, every workspace temporarily loses its live connection.

Tolerable for the prototype: the host keeps the project state locally and browsers reconnect when the relay returns. A production version would run redundant relay nodes so live rooms can survive one node failing.

Relay owns

  • workspace membership
  • forwarding, presence, history delivery
  • reconnect handling

Relay never owns

  • files or commands
  • agent decisions
  • publish rights or source-of-truth state
