Collaborative Local AI Coding

The host laptop is the source of truth. Remote collaborators connect through a thin relay; the host app manages shared workspaces, agent runs, safety boundaries, and every publish to main.

Overview

How it fits together

  • A work session is the shared workspace: one file tree, one draft of work, and one safe place for agents to run. Multiple conversations can collaborate inside it.
  • A conversation is the user-facing chat lane: it has its own transcript and agent memory, but it can still work in the same shared workspace.
  • Clients never touch the host repo or terminal directly. The relay carries messages; the host app decides what can run and what can change.
  • Agents stay aware of each other through short activity updates and host-provided change notices, so they can see when another agent may affect the same files.
  • Publishing is a gated step: the host checks the changes, protects main, and serializes final merges.
Architecture

Components & trust boundary

Shared workspaces

Multiple agents in one workspace

The UI shows active work sessions. A user can resume an existing conversation or start another conversation inside the same workspace. Every agent sees the same file tree, so the product keeps each conversation aware of nearby work.

workspace

Shared working state

One work session has one file tree and one draft of work. When several conversations join it, they are collaborating in the same place, not in copies.

conversation

Separate chat context

Each conversation keeps its own transcript and agent memory. Starting another conversation does not create another workspace.

awareness

Coordination, not polling

Agents receive short updates about what others are doing and what changed on main. They can add their own concise updates before important edits.

Flow

How work moves through the system

Relay

The relay server

The relay is the live transport layer. Browsers connect to it from the LAN, and the host app keeps one long-lived connection open. The relay groups people by workspace, carries presence and chat events, and forwards agent updates back to the right conversations.

The relay does not own project state. If it restarts, users reconnect and the host app restores the visible history. File access, agent actions, and publish decisions stay on the host laptop.

⚠ Single point of failure

One relay node is exactly that — if it goes down, every workspace temporarily loses its live connection.

Tolerable for the prototype: the host keeps the project state locally and browsers reconnect when the relay returns. A production version would run redundant relay nodes so live rooms can survive one node failing.

Relay owns

  • workspace membership
  • forwarding, presence, history delivery
  • reconnect handling

Relay never owns

  • files or commands
  • agent decisions
  • publish rights or source-of-truth state
