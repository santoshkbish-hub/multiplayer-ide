Multiplayer IDE — flows

Six sequence diagrams that cover how sessions live and die, how people and chats join them, how multiple agents work side-by-side on the same sandbox, and how separate sessions share a repo via publish.

1.Vocabulary

The diagrams use these terms consistently. Everything else is an implementation detail.

Session
A unit of collaborative work. Owns one git branch, one worktree, and one sandbox.
Sandbox
A per-session container that all file edits and shell commands run inside. Spawned on demand, destroyed when idle.
Worktree
A checked-out copy of the repo on a session-specific branch (collab/session-…). The sandbox sees this as /work.
Chat
A thread of prompts inside a session. One session can hold many chats; each gets its own Claude memory but they share the worktree and sandbox.
Owner / Joinee
Owners can prompt the agent and publish. Joinees see everything and can chat, but can't drive the agent until ownership is delegated to them.
Publish
Merging a session's branch into main. Other sessions on the same repo fast-forward to the new main so they stay in sync.

2.Session lifecycle — spawn, idle, reap

A session is created when a user opens the IDE. Its sandbox exists only while at least one tab is connected. Five minutes after the last tab closes, the sandbox is destroyed; the session row and git branch stay on disk so it can be resumed later.

sequenceDiagram
  autonumber
  actor User
  participant UI as Browser UI
  participant Daemon
  participant Sandbox
  participant Git as Git worktree

  User->>UI: open the app
  UI->>Daemon: create session for user
  Daemon->>Git: cut branch + checkout worktree
  Daemon->>Sandbox: spawn container, mount worktree as /work
  Daemon-->>UI: session id and invite
  UI->>Daemon: open live connection
  Note over Daemon: session is live, idle timer paused

  User->>UI: close the tab
  UI--xDaemon: connection drops
  Note over Daemon: zero live tabs, arm 5 min idle timer

  alt user comes back within 5 min
    User->>UI: reopen the app
    UI->>Daemon: reconnect
    Note over Daemon: cancel idle timer, reuse same sandbox
  else 5 min elapse with no one connected
    Daemon->>Sandbox: stop and remove container
    Daemon->>Daemon: mark session inactive
    Note over Git: branch and worktree stay on disk
  end

  opt user reopens later
    User->>UI: reopen
    UI->>Daemon: reconnect
    Daemon->>Sandbox: spawn fresh container on the same worktree
    Note over Daemon: session is live again
  end

Idle window is configurable via COLLAB_IDLE_MS. Sessions are never auto-deleted — ending a session is an explicit action.

3.Joining a session

There is no invite step in the prototype. Anyone on the LAN who opens the app picks a name, sees the list of live sessions, and joins whichever one they want — including ones they didn't start. The UI gets fresh credentials minted on the spot and starts streaming the session's chat and activity.

sequenceDiagram
  autonumber
  actor User as New user
  participant UI as Browser UI
  participant Daemon

  User->>UI: pick a name
  UI->>Daemon: list live sessions
  Daemon-->>UI: sessions
  User->>UI: click a session in the list
  UI->>Daemon: join this session as this name
  Daemon-->>UI: credentials and a chat id (existing if user had one here, else new)
  UI->>Daemon: open live connection
  Daemon-->>UI: replay of chat history and recent activity
  Daemon-->>UI: live stream of new chat, file changes, agent tokens

  Note over UI,Daemon: at the same time every other tab in this session
sees a presence update for the new arrival

There is one rule at chat granularity: each chat thread belongs to the first user who claimed it. A different user joining the same session can resume any chat they already own, or start a new one — but they can't take over someone else's existing chat.

4.A new chat inside an existing session

One session can hold several parallel chats. They all share the same worktree and sandbox, but each chat has its own conversation history with the agent — switching chats doesn't drag context across.

  actor User
  participant UI as Browser UI
  participant Daemon
  participant Sandbox
  participant Agent as Claude

  Note over User,Agent: session already exists with chat A in progress

  User->>UI: start new chat in this session
  UI->>Daemon: open new connection with new chat id
  Daemon-->>UI: empty replay, fresh chat
  Note over Daemon,Sandbox: same sandbox, no new container

  User->>UI: send prompt in chat B
  UI->>Daemon: prompt with chat id B
  Daemon->>Agent: run prompt, resume memory for B only
  Agent->>Sandbox: read and edit files in /work
  Agent-->>Daemon: streamed tokens
  Daemon-->>UI: agent reply for chat B

  Note over Agent: chat A and chat B see
each other's file edits via the shared worktree,
but not each other's conversation

5.Many agents collaborating in one session

Multiple chats (or multiple humans driving them) can prompt the same sandbox at the same time. A shared status feed makes each agent aware of what the other just did, so they don't trample each other's work.

sequenceDiagram
  autonumber
  actor A as Alice
  actor B as Bob
  participant UI_A as Alice's chat
  participant UI_B as Bob's chat
  participant Daemon
  participant Feed as Shared status feed
  participant Sandbox
  participant Worktree

  Note over A,B: both connected to the same session

  A->>UI_A: prompt - add login button
  UI_A->>Daemon: agent prompt, chat A
  Daemon->>Feed: log - Alice starting work on auth
  Daemon->>Sandbox: run Claude on chat A
  Sandbox->>Worktree: edit src/Login.tsx
  Daemon->>Feed: log - edit src/Login.tsx
  Daemon-->>UI_A: streamed reply

  B->>UI_B: prompt - tidy the header
  UI_B->>Daemon: agent prompt, chat B
  Daemon->>Feed: read recent feed entries
  Note over Daemon,Feed: Bob's agent sees: Alice just edited Login.tsx
  Daemon->>Sandbox: run Claude on chat B,
inject feed snippet into context
  Sandbox->>Worktree: edit src/Header.tsx
  Daemon->>Feed: log - edit src/Header.tsx
  Daemon-->>UI_B: streamed reply

  Note over A,B: both chats also see live file change events,
so each UI updates as the other agent saves

Any user in the session can drive prompts — the UI gives every joinee owner-level credentials. All chats share the same worktree and sandbox, so file edits show up everywhere, but the agent's conversation memory is per chat. The shared status feed is what keeps the agents loosely aware of one another. (A daemon-level "transfer ownership" admin endpoint exists, but is not surfaced in the UI; it's only reachable from scripts/admin.sh delegate on the host.)

6.Cross-session collaboration on the same repo

Different sessions in the same repo work on separate branches. When one session publishes, the change lands on main and every other live session fast-forwards. The other agents are told that main moved so they can adapt their plans.

sequenceDiagram
  autonumber
  participant SA as Session A worktree
  participant Daemon
  participant Main as main branch
  participant SB as Session B worktree
  participant AgentB as Session B agent

  Note over SA,SB: both sessions started from the same main

  SA->>SA: agent edits files
  SA->>Daemon: publish
  Daemon->>Daemon: validate paths, run checks
  Daemon->>Main: merge session A branch
  Daemon-->>SA: published

  Note over Daemon,Main: main has advanced

  Daemon->>SB: fast-forward worktree to new main
  alt fast-forward succeeds
    Daemon-->>SB: files updated
    Note over AgentB: next prompt sees a note -
main moved, these files changed
  else B has its own work in progress
    Note over SB: stay on B's branch,
publish path will detect this later as stale
  end

  AgentB->>SB: read updated files before continuing

A session that can't fast-forward (because it already has its own commits on top) is flagged stale; the user resolves by publishing or rebasing.
